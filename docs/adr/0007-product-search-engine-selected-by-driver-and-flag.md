# Product search backend is selected per-request by DB driver and a Meilisearch flag, not fixed

A Buyer's free-text search on `ProductsPage` is served by one of three interchangeable engines — `LikeProductSearchEngine`, `FullTextProductSearchEngine`, or `MeilisearchProductSearchEngine` — chosen at request time by `ProductSearchEngineFactory`: if `FTS_ENABLED` is true, Meilisearch is used regardless of database; otherwise MySQL uses its native FULLTEXT index and SQLite falls back to `LIKE`. We chose this over standardizing on one mechanism because the three environments in this codebase already diverge (`.env` runs MySQL, `.env.example`/local dev often runs SQLite, and the Pest suite forces SQLite `:memory:` per `phpunit.xml`) — SQLite has no support for Laravel's `whereFullText()` or MySQL's FULLTEXT indexing at all, so a single mechanism would mean either crippling SQLite dev/test or never getting FULLTEXT's benefit in MySQL production. The `FTS_ENABLED` flag exists as an independent axis (not gated behind the MySQL check) so Meilisearch can be exercised locally against SQLite too, ahead of being turned on anywhere else.

To keep the three engines behaviorally consistent rather than silently diverging, two scope decisions were made deliberately narrow:
- Free-text matching covers only `name` and `description` — not `brand.name`/`category.name` (unlike the Filament admin product table, which does search those). MySQL FULLTEXT can't `MATCH()` across a join in one index, and Meilisearch would need those fields denormalized into the search document; both are legitimate future extensions, not included here.
- No relevance-ranked sort was introduced, even though Meilisearch and MySQL FULLTEXT both support it. The existing `sort` dropdown (`latest`/`price`) stays authoritative whether or not a search term is active, identically across all three engines — LIKE has no ranking signal to offer, and a relevance option that only works on two of three engines would be its own silent trap.
- The minimum search term length is fixed at 3 characters (not 2), matching InnoDB's default `innodb_ft_min_token_size` — below that, MySQL FULLTEXT wouldn't index the term at all, while SQLite's `LIKE` would happily match it, producing environment-dependent result sets for the same query.

## Considered options

- **Standardize on `LIKE` everywhere** — simplest, fully portable, but never benefits from FULLTEXT/Meilisearch even where they're available (MySQL prod, or Meilisearch once adopted).
- **Standardize on Meilisearch only (via Scout)** — best relevance/typo-tolerance, but forces a new infra dependency (the Meilisearch service) to be running in every environment including local dev and CI, for a store at this scale that doesn't yet need it.
- **Driver + flag-selected engine (chosen)** — each environment gets the best mechanism it can actually support, with an explicit opt-in path to Meilisearch; the cost is three implementations of the same interface to maintain, and a documented (not automated) testing gap for the two non-default engines.
